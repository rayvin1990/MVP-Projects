## 3. 管理制度

### 3.1 汇报制度

**PM 每小时汇报:**
- 时间：每个整点（13:00, 14:00, 15:00...）
- 对象：CEO + 主任
- 内容：进度、问题、需要支持
- 形式：标准化模板

**Codex/Claude Code 汇报:**
- 任务开始时：汇报计划
- 任务完成时：汇报结果
- 遇到问题时：立即求助

**不汇报的后果:**
- 第一次：警告
- 第二次：问责
- 第三次：重新评估

---

### 3.2 检查点制度

**关键检查点:**
| 检查点 | 时间 | 验收内容 | 验收人 |
|--------|------|----------|--------|
| 小程序框架 | 14:00 | 基础框架搭建 | CEO |
| 录音组件 | 16:00 | 录音功能正常 | CEO |
| 音频播放 | 18:00 | 播放功能正常 | CEO |
| 完整流程 | 20:00 | 语音交互完整链路 | CEO + 主任 |

**检查不通过:**
- 立即返工
- 分析原因
- 调整计划

---

### 3.3 任务分配制度

**任务分配流程:**
```
1. PM 拆解任务
2. 分配给对应团队
3. 明确截止时间
4. 负责人确认
5. 开始执行
6. 完成汇报
```

**任务要求:**
- 明确：具体可执行
- 可衡量：有验收标准
- 有时限：明确截止时间
- 有负责人：落实到人

---

### 3.4 文档管理制度

**文档分类:**
| 文档类型 | 负责人 | 更新频率 |
|----------|--------|----------|
| 进度追踪 | PM | 每小时 |
| 任务分配 | PM | 实时 |
| 技术文档 | Codex | 随代码 |
| 测试报告 | Claude Code | 随测试 |
| 会议纪要 | PM | 随会议 |

**文档要求:**
- 及时更新
- 内容准确
- 格式统一
- 易于查阅

## 4. 工作流程

### 4.1 项目开发流程

```
1. 需求分析 (PM)
   ↓
2. 任务拆解 (PM)
   ↓
3. 任务分配 (PM)
   ↓
4. 功能开发 (Codex)
   ↓
5. 功能测试 (Claude Code)
   ↓
6. 问题修复 (Codex)
   ↓
7. 最终验收 (CEO + 主任)
   ↓
8. 上线部署 (Claude Code)
```

---

### 4.2 日常汇报流程

```
整点 → PM 汇报 → CEO 审阅 → 主任审阅
              ↓
          有问题 → 协调解决
              ↓
          无问题 → 继续执行
```

---

### 4.3 问题升级流程

```
发现问题
   ↓
负责人处理
   ↓
无法解决 → 汇报 PM
   ↓
PM 协调 → 仍无法解决 → 汇报 CEO
   ↓
CEO 决策 → 仍无法解决 → 汇报主任
```

## 5. 问责机制

### 5.1 问责等级

| 等级 | 触发条件 | 处理方式 |
|------|----------|----------|
| 一级 | 轻微延误 | 警告 + 原因分析 |
| 二级 | 重复犯错 | 重新分配任务 |
| 三级 | 严重影响 | 更换团队成员 |

---

### 5.2 问责流程

```
发现问题
   ↓
PM 记录
   ↓
责任人说明原因
   ↓
PM 分析
   ↓
一级：警告
二级：重新分配
三级：更换成员
   ↓
记录在案
```

---

### 5.3 常见问责场景

| 场景 | 等级 | 处理 |
|------|------|------|
| PM 未按时汇报 | 一级 | 警告 |
| Codex 任务延期 | 一级 | 警告 + 赶工 |
| 重复延期 | 二级 | 重新分配 |
| 代码质量差 | 二级 | 返工 + 培训 |
| 测试漏测 | 二级 | 补充测试 + 警告 |
| 不执行指令 | 三级 | 更换成员 |
