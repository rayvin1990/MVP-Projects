## 今天遇到的问题

### 问题 1：GitHub Pages 404

**现象：**
```
推送了隐私政策 → 访问 URL → 404 Not Found
```

**原因：**
1. 推送到错误的分支
2. GitHub Pages 设置没保存
3. 构建需要时间（1-5 分钟）

**解决：**
1. 确认文件在正确的分支（main）
2. 检查 Pages 设置（Branch = main）
3. 重新点 Save
4. 等待 2-5 分钟

**下次照做：**
```
1. git push 到 main
2. 访问 Settings → Pages
3. 确认 Branch = main
4. 点 Save
5. 等待 2 分钟
6. 验证 URL
```

---

### 问题 2：分支混乱

**现象：**
```
创建了 master、main、pages 三个分支
不知道哪个是正确的
```

**原因：**
- 没有提前确认 GitHub Pages 用哪个分支
- 临时创建多个分支

**解决：**
- 先检查 Pages 设置，确认用哪个分支
- 只推送到那个分支

**下次照做：**
```
1. 先访问 Settings → Pages
2. 看 Branch 选的是什么
3. 只推送到那个分支
```

---

### 问题 3：隐私政策部署

**现象：**
```
用 saasfly 仓库部署 price-to-hours 的隐私政策
名字不对应，不专业
```

**原因：**
- 想省事，用现有仓库
- 没考虑专业性

**更好的做法：**
- 创建专门仓库 `price-to-hours`
- 代码隔离，更专业

**下次照做：**
```
如果是正式项目：
1. 创建专门仓库
2. 部署隐私政策
3. URL 对应项目名称
```

## 开发流程改进（重要）

**标准流程（以后照做）：**

| 步骤 | 谁做 | 内容 |
|------|------|------|
| Step 1 | 主任 | 描述问题/需求 |
| Step 2 | 我 | 复述理解，确认没误解 |
| Step 3 | 我 | 输出修改计划（文件清单 + 每个文件做什么） |
| Step 4 | 主任 | 确认计划 OK |
| Step 5 | 我 | 生成 patch（实际修改文件） |
| Step 6 | 我 | 运行测试/编译 |
| Step 7 | 我 | 根据错误继续修复 |

**核心原则：**
- 先确认再动手（Step 2-4）
- 整体计划，不是单个改（Step 3）
- 测试验证（Step 6）
- 迭代修复（Step 7）
