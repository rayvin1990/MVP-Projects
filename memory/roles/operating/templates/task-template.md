# 任务模板 - Codex / Claude Code

**版本:** v1.0  
**创建时间:** 2026-03-15  
**用途:** 给 Codex/Claude Code 派发任务的标准化格式

---

## 标准格式

```
# <Team / Agent> 任务 - <任务标题>

**项目:** <项目名>
**负责人:** <Owner Agent / Team Lead>
**优先级:** P0 | P1 | P2
**创建时间:** YYYY-MM-DD HH:mm
**截止时间:** YYYY-MM-DD HH:mm
**状态:** 待开始 | 进行中 | 阻塞 | 已完成

---

## 业务目标
- 一句话说明为什么做这件事
- 明确这是业务链中的哪一环

## 输入
- 已有上下文
- 相关文件路径
- 设计稿 / 接口 / 文档 / 数据来源
- 如果是接续任务，要说明上一步做到哪里

## 预期输出
- 要交付的代码
- 要交付的文档
- 要交付的验证结果
- 结果要写回哪里

## 依赖
- 上游依赖
- 环境变量 / key / 账号
- 需要谁先完成什么
- 没准备好的依赖要直接写明

## 执行步骤
1. 步骤 1
2. 步骤 2
3. 步骤 3

## 验收标准
- 什么叫完成
- 如何验证
- 哪些页面 / 接口 / 文件必须正常
- 是否需要测试、部署、截图、材料

## 回写要求
- 完成后更新 `memory/YYYY-MM-DD.md`
- 如有交付物，写清路径
- 如遇阻塞，必须按下面格式回写：

```
Issue:
Impact:
Blocked By:
Suggested Fix:
Decision Needed:
```

---

## 硬规则

1. **不要给模糊任务**，比如"去看下""优化一下""搞一下这个"
2. **每个任务必须有唯一 owner**
3. **每个任务必须有截止时间和验收标准**
4. **没有依赖说明的任务，容易卡死链路**
5. **如果是业务链任务，要写清楚上一环是谁、下一环是谁**
6. **Codex 更适合接**：研发落地、代码修改、工程接线、文档落地
7. **Claude 更适合接**：测试、部署、审核材料、质量门禁
